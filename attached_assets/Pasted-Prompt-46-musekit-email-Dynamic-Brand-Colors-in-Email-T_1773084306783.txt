Prompt 46: musekit-email — Dynamic Brand Colors in Email Templates
Module Repl
musekit-email (GitHub: SpeckledDarth/musekit-email)

Context
The email package renders 7 transactional email templates (Welcome, Verification, Password Reset, Subscription Confirm, Subscription Canceled, Team Invitation, KPI Report). All emails flow through a shared emailWrapper function in shared.ts that provides the HTML layout and styling.

Currently, the primary brand color is hardcoded as #3b6cff in the defaultBrand object in shared.ts. Dark mode colors are hardcoded as #1a1a2e and #16213e. Since CSS variables don't work in email clients, emails must use inline hex values injected at render time.

The good news: a getBrandSettings() function ALREADY EXISTS in src/lib/supabase.ts — and reports.ts already calls it. The issue is that (a) it queries snake_case keys (app_name, primary_color, etc.) that may not match the CSS Dashboard's palette.primaryColor key, and (b) most template callers don't use it.

Reference: See docs/current/css-variable-contract.md for the complete list of available CSS variables. Note: email templates cannot use CSS variables — they must use inline hex values from the same Supabase source.

The Fix
1. Update getBrandSettings() to Read CSS Dashboard Color
In src/lib/supabase.ts, update the getBrandSettings() function to ALSO query palette.primaryColor and prefer it over the legacy primary_color key:

export async function getBrandSettings(): Promise<BrandSettings> {
  // ... existing defaults ...
  try {
    const supabase = getSupabaseClient();
    const { data } = await supabase
      .from('settings')
      .select('key, value')
      .in('key', [
        'app_name', 'primary_color', 'logo_url', 'support_email', 'website_url',
        'palette.primaryColor',  // ← ADD THIS: CSS Dashboard primary color
      ]);
    if (data && data.length > 0) {
      const settingsMap: Record<string, string> = {};
      for (const row of data) {
        settingsMap[row.key] = row.value;
      }
      return {
        appName: settingsMap['app_name'] || defaults.appName,
        // Prefer palette.primaryColor (CSS Dashboard) over legacy primary_color
        primaryColor: settingsMap['palette.primaryColor'] || settingsMap['primary_color'] || defaults.primaryColor,
        logoUrl: settingsMap['logo_url'] || defaults.logoUrl,
        supportEmail: settingsMap['support_email'] || defaults.supportEmail,
        websiteUrl: settingsMap['website_url'] || defaults.websiteUrl,
      };
    }
  } catch {
    // Fall through to defaults
  }
  return defaults;
}
This ensures emails reflect whatever the admin sets in the CSS Dashboard. The legacy primary_color key still works as a fallback.

2. Update ALL Template Callers to Use getBrandSettings()
Each template's render function should accept an optional brand parameter and use getBrandSettings() when not provided:

Option A (preferred — minimal API change): Update the email sending pipeline to always fetch brand before rendering:

In src/client.ts (or wherever sendEmail calls template render functions), add brand resolution:

import { getBrandSettings } from './lib/supabase';
// Before rendering any template:
const brand = await getBrandSettings();
const html = renderWelcomeEmail(vars, brand);
Option B: Make each render function async and self-fetch:

export async function renderWelcomeEmail(vars: WelcomeVars, brand?: BrandSettings) {
  const resolvedBrand = brand || await getBrandSettings();
  return emailWrapper(content, resolvedBrand);
}
Option A is simpler — fetch once, pass everywhere. The key point is that ALL 7 templates receive the dynamic brand.
